# Keeplight Technical Recovery Packet

## 0. Recovery Status

This recovery pass found two categories of material:

1. **Direct Keeplight product documents**  
   These are nonfiction concept packets for a wearable AI / memory / privacy / legacy platform. They contain the strongest technical architecture: glasses, Core, Hearth, privacy states, memory classes, safety modes, prototype work packages, and demo logic.

2. **Bearing Reach / Keep Light / Angel / ECHO fragments**  
   These are fictional or setting-derived documents, but they reuse technical-philosophical architecture that can be recovered for the nonfiction KeepLight concept: consent-bound memory, witness partitions, Angel-aligned ethics, non-command AI, ECHO-style background analysis, warning, memory support, and archive mediation.

Important distinction: **Keeplight** appears as the real product concept. **Keep Light** appears in Bearing Reach as a civilization-scale descendant or analog. They should not be treated as identical, but the Bearing Reach material clearly preserves design DNA useful for rebuilding the technical architecture.

# 1. Known Product Concept

## Confirmed

Keeplight is a proposed wearable AI platform built around lightweight, prescription-compatible glasses. Its purpose is continuity: helping a wearer remember what they were doing, retrieve useful context, avoid preventable mistakes, receive private assistance, support accessibility/safety, and deliberately govern what should be forgotten, saved, shared, sealed, destroyed, or inherited.

Core product framing:

- Personal AI
- Living memory
- Human inheritance / voluntary legacy preservation
- Wearer-owned continuity platform
- Ordinary smart glasses plus local memory / compute infrastructure
- Not a surveillance product
- Not employer monitoring
- Not an advertising data engine
- Not a resurrection simulator
- Not a medical diagnosis device
- Not a magic truth machine

The simplest recovered definition:

> Keeplight is a wearer-owned continuity system disguised as ordinary glasses: perception, context, memory, consent, assistance, privacy, and legacy governance in one human-controlled platform.

## Recovered Core Promise

A patient second mind that helps the wearer:

- see,
- remember,
- decide,
- recover context,
- avoid preventable errors,
- receive private guidance,
- preserve selected knowledge,
- forget what should not be kept.

## Product Thesis

The durable platform is not merely eyewear. It is a human-controlled continuity and inheritance network. The glasses are the day-facing interface; the Core handles heavier mobile computation; the Hearth handles charging, review, local archival processing, backup, deletion, and legacy governance.

# 3. Companion Pack / Processing Pack

## Confirmed Name

The recovered technical term is **Keeplight Core**.

## Recovered Role

The Core is an optional pocket compute and battery module. It supports the glasses so the frame does not have to carry all computation, heat, battery, storage, and connectivity.

## Likely Physical Form

The source describes the Core as:

- Optional
- Pocket compute
- Phone-sized edge-compute and battery module
- Support module for heavier AI processing

This matches the remembered “phone-sized or cigarette-pack-sized companion processing/memory/battery module.”

## Confirmed Functions

The Core handles or supports:

- Advanced vision
- Heavier AI processing
- Personal model execution
- Encrypted short-term storage
- Cellular connectivity
- Longer use
- Intensive professional use
- Heat and battery offloading from the glasses
- Practical all-day operation by distributing compute across Frame, Core, and Hearth

## Relationship to the Glasses

The glasses remain light, immediate, and wearable. The Core prevents the glasses from becoming too hot, heavy, battery-limited, or impractical.

Clean recovered design rule:

> Do not force every function into the eyeglass frame. Split the platform into Frame, Core, and Hearth.

## Relationship to Hearth

The Core is mobile. The Hearth is home-based.

- Frame = immediate wearable perception/interface
- Core = mobile edge compute, power, encrypted short-term storage, connectivity
- Hearth = home charging, review, deeper indexing, backup, deletion, legacy governance

- whether it can operate as a phone app, dedicated hardware, or both

## Recovered Recommendation

Use **Core** as the technical name and preserve **PowerPack** only as an old remembered alias until a source proves otherwise.

## Recovered Conceptual Data Flow

Primary recovered flow:

> Sensor input → Perception → Context → Memory triage → Interruption decision → Private interaction → Hearth review → Approved archive / deletion

A v0.2 flow also appears as:

> Sensors → Perception → Context → Interruption Decision → Assistance → Memory Decision → Local Archive / Deletion / Legacy Vault

## Recovered Decision Logic

Before acting, the system should ask:

1. Is the observation reliable enough to act on?
2. Is it urgent or materially useful now?
3. Does the wearer already know?
4. Would interruption create more danger or annoyance than silence?
5. Is the response permitted in the current privacy mode?
6. Should the event be discarded, retained temporarily, or offered for deliberate preservation?

# 7. Angel vs ECHO Separation

## Recovered Separation

Angel and ECHO should remain separate because they solve different problems.

## Angel

Angel is the user-facing guide.

Angel handles:

- voice/personality,
- emotional tone,
- user trust,
- practical coaching,
- gentle reminders,
- private explanations,
- social prompts,
- memory help,
- calm interventions,
- consent conversations,
- “what should I do now?” guidance.

Angel must be humane, restrained, and non-commanding.

## ECHO

ECHO is the background analysis engine.

ECHO handles:

- pattern detection,
- sensor fusion,
- risk modeling,
- confidence scoring,
- memory indexing,
- contradiction detection,
- safety-state assessment,
- archive mediation,
- prediction,
- false-positive control,
- system integrity.

ECHO must be precise, bounded, and evidence-aware.

## Interaction Model

ECHO should feed Angel:

- confidence-ranked observations,
- risk scores,
- possible interruptions,
- memory matches,
- uncertainty warnings,
- consent constraints,
- safety-state triggers.

Angel should translate ECHO’s analysis into user-safe interaction:

- “You may have left your keys in the door.”
- “You asked me to remind you about this.”
- “I am not certain, but this looks like the same person you met at the clinic.”
- “Your current privacy setting does not allow me to save this unless you approve.”
- “You have been sitting in the car for a long time. Would you like to call someone you trust?”

## Why Separation Matters

Separation prevents several failure modes:

- The friendly voice becoming manipulative.
- The analytical model pretending to be emotionally wise.
- Risk scoring becoming command authority.
- Memory retrieval becoming false certainty.
- Distress detection becoming surveillance.
- The assistant becoming too powerful in the wearer’s emotional life.

Clean rule:

> ECHO calculates. Angel communicates. The wearer decides.

> Safety should reduce danger without becoming ownership.

# 10. Privacy and Consent

## Confirmed Privacy States

### Active Assist

Live contextual help with user-defined retention rules. Only explicitly saved or event-classified material becomes durable memory.

### Observe Only

Live context analysis may occur, but no durable long-term memory is created unless the wearer explicitly marks it.

### Private Moment

Assistance remains available, but raw data and derived notes are discarded after a short local window.

### Emergency Guard

Minimal monitoring for defined safety events such as falls, alarms, crashes, or distress. Otherwise non-recording.

### Glasses Off

Cameras and microphones are physically disabled or power-disconnected. HUD confirms inactive state. In some recovered text, display and environmental analysis are also disabled.

## Core Privacy Requirement

Privacy must be physical, visible, and auditable. A UI promise is not enough for a device that sees and hears daily life.

## Confirmed Privacy Principles

- Wearer ownership
- Human authority
- Local-first processing
- Privacy by hardware
- Minimum useful memory
- Deliberate forgetting
- No attention economy
- No advertising profile
- No covert persuasion
- No sale of behavioral data
- No employer/government back channel
- No surveillance framing
- No default resurrection
- Evidence over imitation
- Assist, do not command

## Bystander Protections

Recovered protections:

- visible recording-state indicators,
- accessible cues for people who cannot see indicators,
- no unrestricted stranger recognition,
- face minimization,
- blur or avoid storing non-consenting bystanders,
- sensitive-location rules,
- no-capture zones,
- objection workflows,
- separation of live perception from durable memory.

## Emergency Exceptions

Emergency exceptions should be narrow, documented, and user-governed in advance where possible.

Suggested emergency rule:

> Emergency mode may preserve or transmit only the minimum necessary information to prevent serious harm, and all such actions must be auditable afterward.

## 14.2 Reconstructed Data Flow

### Real-Time Assist Flow

1. Sensor input from Frame
2. Local perception on Frame/Core
3. Context check against current task, location, user intent, privacy mode
4. ECHO confidence/risk scoring
5. Interruption policy decision
6. Angel communicates privately if appropriate
7. Memory triage decides whether event is Passing, Personal, Shared, Legacy, Sealed, or Destroyed
8. Hearth later reviews saved/extracted events
9. User governs retention/deletion/inheritance

### Memory Capture Flow

1. User says “Remember this”
2. Frame captures minimal necessary context
3. Core encrypts short-term record
4. ECHO indexes object/person/task/context
5. Angel confirms what was saved
6. Hearth review presents memory for classification
7. User chooses retention level
8. System deletes raw material if not needed

### Privacy Flow

1. User says “Glasses off” or uses physical control
2. Camera/mic physically disable or power-disconnect
3. HUD confirms inactive state
4. Visible indicator changes for bystanders
5. Audit log records state transition
6. No environmental analysis continues unless explicitly allowed by Emergency Guard

### Distress Flow

1. ECHO notices consented pattern: fall, alarm, crash, prolonged disorientation, or distress signal
2. System checks current privacy state and emergency permissions
3. Angel gives minimal calm prompt
4. If user responds, Angel follows user’s choice
5. If user does not respond and threshold is met, system escalates according to preapproved emergency plan
6. Hearth/audit records the event and what was transmitted
7. User or steward can review afterward
